### Dockerfile


## 4. Выявленные проблемы и их статус

### P0: Docker image содержит файлы чужого проекта

**Проблема:** `docker exec constitution_api ls /app/` показывает `content.yaml`, `import_yaml.py`, `templates/` и другой main.py (24KB) — от совершенно другого проекта.

**Причина:** Имя образа `constitution_api` уже было занято другим проектом на VPS. `docker build -t constitution_api .` перезаписал тег, но старый cached layer всё ещё вмешивался.

**Решение:** `docker build --no-cache -t constitution_api .` из `/root/constitution-rag/` + предварительно `docker rm -f constitution_api`.

**Статус:** ❌ Ожидает выполнения пользователем.

**Команды для исправления:**
```bash
cd /root/constitution-rag
git pull origin main
docker rm -f constitution_api
docker build --no-cache -t constitution_api .
docker run -d --name constitution_api \
  --restart unless-stopped \
  --network host \
  --env-file .env \
  constitution_api

# Проверка: должны быть НАШИ файлы
docker exec constitution_api ls /app/app/
# Ожидаем: answer_runner.py  ask_cli.py  db.py  retrieval_runner.py

# Проверка health
curl http://localhost:8000/health

# Тест API
curl -s -X POST http://localhost:8000/api/ask \
  -H 'Content-Type: application/json' \
  -d '{"query":"Что говорится о свободе слова?"}' | python3 -m json.tool
```

### P1: Нет .dockerignore

**Проблема:** `COPY . .` в Dockerfile копирует всё: `.venv/`, `.git/`, `raw/` (PDF ~15MB), `normalized/`. Docker build context = 154MB.

**Решение:** Создать `.dockerignore`:
```
.venv/
.git/
raw/
normalized/
__pycache__/
*.pyc
.env
.DS_Store
*.log
tmp/
qa/evidence/
docs/
```

**Статус:** ❌ Нужно создать и закоммитить.

### P2: docker-compose.yml в /root/ — ловушка

**Проблема:** Если кто-то запустит `docker compose up` из `/root/constitution-rag/`, это создаст ВТОРОЙ PostgreSQL и попытается создать контейнер `constitution_api` (конфликт имён).

**Решение:** Добавить warning в docker-compose.yml и README (README уже обновлён).

**Статус:** ⚠️ Warning есть в README, но не в самом docker-compose.yml.

### P3: requirements.txt дублирует deps в Dockerfile

**Проблема:** `requirements.txt` содержит `fastapi[standard]>=0.115` и `uvicorn[standard]>=0.32`, но Dockerfile ТАКЖЕ делает `pip install "fastapi[standard]>=0.115" uvicorn[standard]`. Дублирование.

**Решение:** Убрать из Dockerfile отдельный pip install fastapi/uvicorn — они уже в requirements.txt.

**Статус:** 🟡 Не блокер, cosmetic.

### P4: .gitignore не содержит raw/ и normalized/

**Текущий .gitignore:**
```
.venv/
__pycache__/
*.pyc
.env
.DS_Store
*.log
tmp/
```

**Замечание:** `raw/` и `normalized/` трекаются в git (суммарно >100MB). Это осознанный выбор для reproducibility, но может быть проблемой при масштабировании. Оставляем как есть.

## 7. Открытые вопросы — ответы

### Q1: Почему `docker exec` показывал чужие файлы?

**Ответ:** На VPS уже существовал Docker-образ с тегом `constitution_api` от другого проекта. Когда мы сделали `docker build -t constitution_api .`, Docker использовал кэшированные слои от старого образа. Решение: `--no-cache`.

### Q2: Почему `/api/ask` возвращал 500, если venv напрямую работает?

**Ответ:** Внутри контейнера находился `main.py` от другого проекта (24KB, с `content.yaml`, `import_yaml.py`). Этот чужой `main.py` не мог импортировать `app.answer_runner`, поэтому запросы падали. venv на хосте работал с правильным кодом из `/root/constitution-rag/`.

### Q3: Почему `docker compose down` из `/root/` убил базу?

**Ответ:** `docker compose down` удаляет ВСЕ контейнеры из compose-файла. Поскольку `docker-compose.yml` в `/root/constitution-rag/` описывает и `api`, и `db` сервисы, команда попыталась остановить оба. Контейнер `constitution_rag_db` живёт в `/opt/`, но docker compose вызывается по имени контейнера, а не по пути.

**Уточнение:** На самом деле `docker compose down` из `/root/` не удалил `/opt/`-шный контейнер, потому что compose-проекты изолированы. Проблема была в том, что при запуске `docker compose up` из `/root/` был создан ВТОРОЙ PostgreSQL-контейнер (с именем `constitution_db`, не `constitution_rag_db`), и `docker compose down` убил именно его. А оригинальный `constitution_rag_db` из `/opt/` уже не был запущен (возможно, был остановлен ранее).

### Q4: Где живёт .env на VPS?

**Ответ:** `/root/constitution-rag/.env` — для API-контейнера (через `--env-file .env`).
`/opt/constitution-rag/.env` — для DB-контейнера (через docker-compose `env_file`).

### Q5: Почему 45 тестов, если файл называется run_full30.py?

**Ответ:** Исторически было 30 тестов (ru). В batch-3 добавлены RT-31..38 (kz) и RT-39..45 (en). Имя файла сохранено для обратной совместимости. В docstring написано "Full 30+15 red-team QA run".

### Q6: Зачем `--network host` для API-контейнера?

**Ответ:** Потому что PostgreSQL слушает на `127.0.0.1:55432` хоста. Если бы API был в Docker bridge network, он не смог бы достучаться до `127.0.0.1:55432`. С `--network host` контейнер видит все порты хоста напрямую.

### Q7: Можно ли перейти на Docker network вместо `--network host`?

**Ответ:** Да, но потребуется:
1. Подключить API к той же Docker network, что и БД (`constitution-rag_default` из `/opt/constitution-rag/`)
2. Изменить `DB_HOST` в `.env` на имя контейнера (`constitution_rag_db`)
3. Изменить `DB_PORT` на `5432` (внутренний порт PostgreSQL)

Пока `--network host` проще и работает.

### Q8: Что с Nginx?

**Ответ:** `constitution_nginx` слушает на порту `8080` и проксирует на `127.0.0.1:8000` (FastAPI). Конфиг nginx не в нашем репозитории — он от другого/стороннего развёртывания. Публичный доступ: `http://srv1380923.hstgr.cloud:8080/`.

### Q9: Нужен ли `fastapi[standard]` И в requirements.txt, И в Dockerfile?

**Ответ:** Нет, это дублирование. fastapi и uvicorn уже в `requirements.txt`. Строка в Dockerfile `pip install "fastapi[standard]>=0.115" uvicorn[standard]` — лишняя. Не ломает ничего, но засоряет.
