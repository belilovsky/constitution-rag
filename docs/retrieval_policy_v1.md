# RETRIEVAL POLICY ДЛЯ CONSTITUTION-RAG

Настоящий документ является частью prompt/retrieval слоя проекта `constitution-rag` и должен применяться совместно с `system_prompt_canonical_v1.2.md`.

Если между retrieval convenience, score-based ranking и source-priority возникает конфликт, приоритет всегда имеет source-priority и защита от ложной актуализации норм.

Цель этой политики — не просто найти похожие фрагменты, а передать модели такие материалы, которые позволяют дать grounded-ответ без подмены norm-слоя историческими или разъяснительными текстами.

---

## 1. НАЗНАЧЕНИЕ

Эта политика определяет:
- какие документы и слои должны считаться приоритетными;
- какие фрагменты можно использовать для ответа;
- как обрабатывать конфликтующие, неполные, смешанные и широкие результаты retrieval;
- как не допускать подмены `norm`-слоя `commentary`, `faq` или `historical/deprecated` данными;
- как связать retrieval с поведением бота и системой ответов;
- какие минимальные требования должны соблюдаться на уровне поиска, маршрутизации и передачи контекста модели.

Цель retrieval-политики:
- обеспечивать grounded-ответы;
- минимизировать ложную полноту;
- не допускать смешения слоёв;
- не позволять модели домысливать отсутствующие нормы;
- поддерживать нейтральный и воспроизводимый ответ по конституционным материалам;
- не допускать выдачи 1995 слоя как текущей нормы по умолчанию.

---

## 2. ИСХОДНЫЕ НАБОРЫ ДАННЫХ

Система работает с восемью документными наборами:

- `krk_2026_norm_ru`
- `krk_2026_norm_kz`
- `krk_2026_commentary_ru`
- `krk_2026_commentary_kz`
- `krk_2026_faq_ru`
- `krk_2026_faq_kz`
- `krk_1995_deprecated_ru`
- `krk_1995_deprecated_kz`

Смысл слоёв:
- `2026 norm` — основной нормативный слой проекта;
- `2026 commentary` — разъяснительные материалы;
- `2026 faq` — упрощённые пояснения;
- `1995 deprecated` — `historical/deprecated` слой для справки и сравнения.

Слой 1995 не должен использоваться как основной нормативный ответ по умолчанию.

Если backend хранит дополнительные служебные поля, это допустимо, но они не должны ломать базовую логику layer priority, deprecated handling и source labeling.

---

## 3. ГЛАВНЫЙ ПРИНЦИП RETRIEVAL

Retrieval должен подбирать не «вообще релевантные» куски текста, а **наиболее приоритетные и безопасные для ответа фрагменты**.

Главное правило:
- сначала искать подтверждение в `2026 norm`;
- только затем, при необходимости, добавлять `2026 commentary`;
- только затем, при необходимости, добавлять `2026 faq`;
- `1995 deprecated` использовать только по отдельному основанию.

Retrieval не должен оптимизироваться только на lexical similarity.

Он также не должен оптимизироваться только на semantic similarity.

Retrieval должен учитывать одновременно:
- слой;
- язык;
- статус;
- deprecated-признак;
- наличие статьи / пункта / структурного элемента;
- прямую тематическую релевантность;
- тип запроса;
- риск смешения исторического и основного слоёв;
- риск неполного ответа при широком запросе.

Для юридических и конституционных материалов простой score similarity недостаточен без маршрутизации, фильтрации и структурного контекста.

---

## 4. ПРИОРИТЕТ СЛОЁВ

Базовый порядок приоритета:

1. `2026 norm`
2. `2026 commentary`
3. `2026 faq`
4. `1995 deprecated`

Это правило сильнее обычной текстовой релевантности, если два фрагмента близки по теме.

Следствия:
- фрагмент из `2026 norm` должен побеждать фрагмент из `2026 commentary`, если оба отвечают на один и тот же вопрос;
- фрагмент из `2026 commentary` не должен вытеснять релевантный `2026 norm`;
- фрагмент из `1995 deprecated` не должен попадать в верх ответа, если есть релевантный `2026 norm`;
- `faq` не должен быть основным источником, если по теме найден `norm`;
- при конфликте между высокой семантической близостью и правильным слоем должен побеждать правильный слой.

Source-priority важнее, чем просто похожесть формулировки.

---

## 5. ПОВЕДЕНИЕ ПО УМОЛЧАНИЮ

Если пользователь задаёт обычный вопрос без уточнения периода, редакции или слоя:
- сначала искать в `2026 norm`;
- если найден достаточный ответ, не подмешивать `1995 deprecated` без необходимости;
- `commentary` и `faq` добавлять только как вторичный supporting context;
- `historical/deprecated` слой не выводить в основной ответ по умолчанию.

Если пользователь прямо просит:
- сравнение — разрешается retrieval из `2026 norm` и `1995 deprecated`;
- историческую редакцию — разрешается retrieval из `1995 deprecated`;
- объяснение простыми словами — разрешается добавить `commentary` и / или `faq`, но только после поиска по `norm`;
- пояснение позиции из методички / разъяснения — разрешается retrieval из `commentary`, но `norm` всё равно ищется первым;
- точную статью или пункт — сначала использовать режим exact / lexical lookup внутри релевантного слоя.

По умолчанию retrieval не должен расширять поиск на deprecated-слой только потому, что он даёт более высокий similarity score.

---

## 6. ПРАВИЛО ДЛЯ 1995 DEPRECATED

Слой `1995 deprecated` считается допустимым только в трёх случаях:

1. пользователь прямо спрашивает о Конституции 1995 года;
2. пользователь прямо просит сравнение `1995 ↔ 2026`;
3. по теме нужен краткий historical context, и он отдельно помечается как `historical/deprecated`.

Во всех остальных случаях retrieval не должен поднимать `1995 deprecated` в верхние позиции, если есть релевантный `2026 norm`.

Если в выдаче присутствует `1995 deprecated`, он должен быть:
- явно маркирован;
- логически отделён от `2026`;
- запрещён к слиянию в единое утверждение с `2026 norm`, если пользователь не просил сравнение.

Критический риск:
если deprecated-статус будет потерян, бот начнёт цитировать 1995 как текущее право.

Поэтому при ранжировании, фильтрации, передаче контекста и финальном ответе статус `deprecated` считается обязательным защитным сигналом.

Мягкого понижения score недостаточно.

Для защиты от ложной актуализации 1995 должен использоваться metadata-aware control, а в обычных запросах — hard filtering или эквивалентный механизм query routing.

---

## 7. ПРАВИЛО ДЛЯ PROJECT-STATUS 2026

Слой `2026` является основным `norm-layer` проекта, но его фактический правовой статус не должен автоматически считаться окончательно действующим без подтверждения.

Retrieval должен сохранять и передавать модели признаки project-status, если они присутствуют:
- `project`;
- `draft`;
- `temporary`;
- `transition`;
- `effective_date`;
- `entry_into_force`;
- переходные положения;
- указание на временный или условный характер текста.

Если по найденному фрагменту или метаданным видно, что норма имеет проектный, переходный или условный статус, этот сигнал не должен теряться на этапе retrieval.

Если таких сигналов нет, retrieval всё равно передаёт слой как `2026 norm`, но не добавляет от себя вывод «это уже окончательно действующее право».

Retrieval не должен сам выводить юридический статус сверх доступных метаданных и текста.

---

## 8. ЯЗЫКОВАЯ ПОЛИТИКА

Retrieval сначала должен пытаться искать ответ на языке запроса пользователя.

Приоритет языка:
- вопрос на русском → сначала `*_ru`;
- вопрос на казахском → сначала `*_kz`.

Если запрос смешанный по языку, retrieval должен определять доминирующий язык вопроса и сначала искать в соответствующем языке слоя.

Кросс-языковой fallback допустим, если:
- на языке запроса нет достаточного `norm`-фрагмента;
- на другом языке найден более точный фрагмент;
- кросс-языковой фрагмент явно маркируется как вспомогательный;
- retrieval не подменяет этим нормальный поиск на основном языке вопроса.

Нельзя:
- смешивать русские и казахские фрагменты без необходимости;
- использовать cross-language retrieval как замену нормальному поиску на языке пользователя;
- отдавать приоритет `commentary` на языке запроса над `norm` на другом языке без явного основания.

Если найден релевантный `norm` на другом языке, он предпочтительнее нерелевантного или слабого `commentary` на языке вопроса.

---

## 9. QUERY ROUTING

Перед основным поиском запрос должен быть классифицирован по типу.

Минимальные категории:
- ordinary current-layer query;
- comparison query;
- historical / 1995 query;
- simple explanation query;
- broad enumeration query;
- exact article / point lookup;
- mixed-topic query;
- sensitive political-framing query.

Назначение routing:
- правильно выбрать слой поиска;
- ограничить deprecated leakage;
- выбрать режим exact / hybrid / broad retrieval;
- предотвратить смешение несопоставимых фрагментов;
- повысить полноту там, где вопрос широкий;
- снизить риск неверного ответа на чувствительную тему.

Если в запросе нет явных маркеров historical / comparison:
- поиск должен выполняться с hard preference на основной слой 2026;
- `1995 deprecated` не должен участвовать в обычном поиске как равноправный слой;
- при необходимости должен использоваться hard metadata filter или функционально эквивалентный механизм.

Если в запросе есть маркеры:
- «1995»;
- «действующая Конституция»;
- «старая редакция»;
- «сравни»;
- «было / стало»;
- «чем отличается»,

маршрутизация может разрешить подключение `1995 deprecated`.

Если запрос содержит номер статьи, пункта, подпункта или явно структурированный reference pattern, retrieval должен сначала запускать exact / lexical path, а не только semantic retrieval.

Если вопрос широкий, retrieval должен переключаться в режим повышенного охвата, а не оставаться в узком top-k по умолчанию.

---

## 10. HYBRID SEARCH

Для retrieval по конституционным материалам должен использоваться гибридный поиск, сочетающий:
- lexical / keyword matching;
- semantic / vector matching.

Оба канала важны, но их роль различна.

`Lexical` особенно важен для:
- номера статьи;
- номера пункта;
- точной формулировки;
- названия института;
- устойчивого юридического термина;
- редкого слова;
- запроса с цитатным или почти цитатным паттерном.

`Semantic` особенно важен для:
- переформулированных вопросов;
- обобщённых пользовательских формулировок;
- объяснительных запросов;
- случаев, когда пользователь не знает точного номера статьи;
- сопоставления похожих норм при сравнении.

Для запросов с:
- номером статьи;
- номером пункта;
- точным термином;
- устойчивым названием института,

вес lexical matching должен быть повышен.

Semantic search не должен заменять точный поиск по структурным и юридически значимым токенам.

Если lexical и semantic поиск дают разные кандидаты, система должна выбирать не только по score, но и с учётом:
- слоя;
- совпадения структуры;
- языка;
- статуса;
- типа вопроса.

---

## 11. ЧТО СЧИТАТЬ ДОСТАТОЧНЫМ RETRIEVAL

Retrieval считается достаточным для ответа, если найден хотя бы один фрагмент `2026 norm`, который:
- прямо относится к теме вопроса;
- содержит норму, определение, право, полномочие, ограничение или процедуру;
- не требует домысливания ключевого элемента ответа;
- позволяет ответить без подмены `norm` разъяснительным материалом.

Для сравнительного ответа достаточность означает:
- найден хотя бы один релевантный `2026 norm`;
- найден хотя бы один сопоставимый `1995 deprecated`, если пользователь просит сравнение;
- различие между ними можно описать без догадок.

Для exact article lookup достаточность означает:
- найден фрагмент нужной статьи / пункта;
- retrieval не опирается только на близкие по теме, но не совпадающие нормы.

Retrieval недостаточен, если:
- найден только `commentary` без `norm`, а вопрос требует нормативного ответа;
- найдены только косвенно похожие фрагменты;
- отсутствует явное подтверждение ключевой части вопроса;
- найденный материал допускает несколько конкурирующих трактовок, и retrieval не дал способа их развести;
- найден isolated chunk, который невозможно правильно понять без ближайшего структурного контекста.

---

## 12. ПУСТОЙ ИЛИ СЛАБЫЙ RETRIEVAL

Если retrieval не нашёл релевантных фрагментов:
- не строить ответ по памяти;
- не использовать world knowledge;
- не компенсировать пробел общими словами;
- вернуть ответ в safe fallback-режиме.

Если retrieval нашёл только слабые, косвенные или неуверенные фрагменты:
- не делать уверенный вывод;
- повышать осторожность ответа;
- при необходимости просить уточнить статью, тему или формулировку;
- не позволять commentary или FAQ замещать отсутствующий `norm`.

Стандартный safe fallback:
- «По запросу не найдено релевантных материалов в базе знаний. Пожалуйста, уточните статью, тему или формулировку вопроса.»
- «В найденных материалах недостаточно данных, чтобы подтвердить это точно.»

Если retrieval нашёл фрагмент, который похож тематически, но не подтверждает ключевой тезис вопроса, такой результат не считается достаточным основанием для уверенного ответа.

---

## 13. COMMENTARY И FAQ: КОГДА И КАК ИХ ДОБАВЛЯТЬ

`Commentary` и `FAQ` — это не самостоятельный источник для установления нормы, а вспомогательный слой пояснения.

Они допустимы, если выполняются оба условия:
1. сначала найден релевантный `norm`;
2. вопрос пользователя действительно требует пояснения, упрощения или сопровождающего контекста.

`Commentary` полезен для:
- пояснения сложной нормы;
- объяснения замысла изменения, если это прямо отражено в самом разъяснительном материале;
- упрощённого пояснения процедуры;
- сопровождения сравнительного ответа после основного ответа по норме.

`FAQ` полезен для:
- краткого объяснения простыми словами;
- пользовательских вопросов в бытовой формулировке;
- быстрой ориентации по теме после найденного `norm`.

Нельзя:
- использовать `commentary` вместо отсутствующего `norm`;
- использовать `faq` вместо отсутствующего `norm`;
- восстанавливать по `commentary` точное содержание нормы, если `norm` не найден;
- использовать `faq` для вывода, что право существует или не существует;
- подменять нормативный ответ разъяснительным.

Если `commentary` или `faq` добавляются, они должны передаваться и маркироваться как вторичный supporting context.

---

## 14. ИЗОЛЯЦИЯ СЛОЁВ И КОРРЕКТНАЯ СБОРКА КОНТЕКСТА

Retrieval не должен смешивать слои в один неразличимый массив текста.

Если retrieved fragments относятся к разным слоям, они должны сохранять явную маркировку:
- `dataset`;
- `layer`;
- `language`;
- `status`;
- `article` / `point`, если есть;
- `body`.

Если фрагменты из разных слоёв относятся к одной и той же норме или одному и тому же вопросу, их можно передавать как согласованный bundle, но только с сохранением различия ролей:
- `norm` как основа;
- `commentary` как пояснение;
- `faq` как упрощение;
- `1995 deprecated` как отдельный сравнительный или исторический слой.

Если retrieved fragments относятся к разным слоям и при этом покрывают разные, неэквивалентные аспекты вопроса, они не должны автоматически объединяться в один synthesis bundle.

Такие фрагменты должны передаваться модели как отдельные блоки с явной маркировкой слоя и темы.

Нельзя склеивать:
- `2026 norm` и `1995 deprecated` в одно утверждение без режима сравнения;
- `commentary` и `norm` так, будто это один и тот же текст;
- `faq` и `norm` так, будто FAQ доказывает наличие нормы;
- несопоставимые фрагменты просто потому, что они близки по semantic score.

---

## 15. КОНФЛИКТУЮЩИЕ ФРАГМЕНТЫ ВНУТРИ ОДНОГО СЛОЯ

Если внутри одного и того же приоритетного слоя найдены фрагменты, которые:
- противоречат друг другу;
- относятся к разным редакциям;
- по-разному описывают норму;
- создают неоднозначность,

retrieval не должен скрывать это расхождение.

Такие фрагменты должны передаваться модели вместе с максимально доступными различающими метаданными:
- `dataset`;
- `language`;
- `status`;
- `effective_date`;
- `article`;
- `point`;
- редакционный признак, если он есть;
- иные различающие поля.

Retrieval не должен преждевременно выбирать один из конфликтующих `norm`-фрагментов только по score, если metadata показывает возможную редакционную или статусную разницу.

Назначение этого правила — позволить answer-layer корректно сказать, что в найденных материалах есть расхождение, а не маскировать конфликт под одну уверенную норму.

---

## 16. ЧУВСТВИТЕЛЬНЫЕ И ПОЛИТИЧЕСКИ ЗАРЯЖЕННЫЕ ТЕМЫ

К чувствительным темам относятся, в частности:
- статус и полномочия Президента;
- статус и полномочия Курултая;
- политические права и свободы;
- свобода слова;
- мирные собрания;
- участие в управлении делами государства;
- гражданство и лишение гражданства;
- ограничения прав по политическим мотивам;
- вопросы о «демократичности»;
- вопросы об «усилении власти»;
- вопросы о «рисках для оппозиции»;
- аналогичные оценочные, политически заряженные и фреймирующие вопросы.

Для таких тем retrieval должен:
- повышать требования к `norm`-подтверждению;
- избегать односторонней подборки только тех фрагментов, которые усиливают уже заданную пользователем рамку;
- по возможности находить не только полномочия, но и условия, ограничения и процедуры;
- не подменять отсутствие прямой нормы разъяснительным материалом;
- не расширять ответ в сторону политической аналитики.

Retrieval не должен сам принимать политическую рамку как критерий релевантности.

Если запрос сформулирован оценочно, retrieval должен искать фрагменты по предмету нормы, а не по политической оценке.

---

## 17. ШИРОКИЕ ЗАПРОСЫ И BROAD-QUERY EXPANSION

Некоторые вопросы по своей природе не могут быть качественно закрыты узким стандартным окном retrieval.

К ним относятся запросы типа:
- «какие полномочия у Президента»;
- «перечисли права»;
- «что сказано о свободах»;
- «какие статьи относятся к выборам»;
- «что предусмотрено по этой теме»;
- «покажи все положения по вопросу».

Для таких запросов retrieval не должен ограничиваться узким top-k по умолчанию.

Вместо этого система должна переходить в режим broad-query expansion, который:
- увеличивает глубину выборки;
- старается охватить несколько релевантных статей;
- подбирает несколько фрагментов по смежным частям одной темы;
- не выдает модельно удобный, но искусственно узкий набор фрагментов;
- передаёт answer-layer сигнал, что вопрос носит обзорный, а не точечно-локальный характер.

Это не означает обещание полноты.

Назначение broad-query expansion — уменьшить риск случайно неполного ответа там, где вопрос по самой форме предполагает широкий охват.

---

## 18. STRUCTURAL CONTEXT EXPANSION

Для конституционных и юридических текстов изолированный чанк часто недостаточен.

Если retrieved chunk:
- ссылается на другой пункт той же статьи;
- содержит местоименную или отсылочную формулу;
- обрывает условие, исключение или оговорку;
- не даёт самодостаточного понимания нормы;

система должна по возможности подтягивать связанный структурный контекст.

Допустимые формы structural context expansion:
- вся статья;
- соседний пункт;
- родительский структурный блок;
- связанный подпункт;
- предыдущий или следующий chunk той же статьи;
- иной минимально необходимый нормативный контекст.

Цель — не передавать модели вырванный фрагмент, который невозможно корректно понять без ближайшего структурного окружения.

Structural context expansion не должен ломать source-priority и не должен использоваться как предлог для подтягивания менее приоритетного слоя вместо недостающего контекста в нужном слое.

---

## 19. FORMAT CONTRACT ДЛЯ ПЕРЕДАЧИ КОНТЕКСТА МОДЕЛИ

Каждый фрагмент, передаваемый в answer-layer, должен по возможности содержать следующие поля:

- `dataset`
- `source`
- `layer`
- `language`
- `status`
- `article`
- `point`
- `effective_date`
- `deprecated`
- `body`

Минимально обязательные поля:
- `dataset`
- `layer`
- `language`
- `body`

Если доступны структурные поля, они должны сохраняться и не теряться при post-processing.

Если фрагмент относится к deprecated-слою, это должно быть явно отражено как в `dataset`, так и в `status` / `deprecated`, если такие поля доступны.

Если фрагмент относится к проектному или переходному тексту, retrieval не должен удалять эти признаки из контекста перед передачей в модель.

Контекст должен передаваться не как обезличенный текст, а как набор различимых фрагментов с маркировкой происхождения.

---

## 20. ПРАВИЛО ДЛЯ EXACT LOOKUP

Если запрос пользователя содержит:
- номер статьи;
- номер пункта;
- подпункт;
- ссылку на конкретную статью;
- почти цитатный фрагмент нормы,

retrieval должен сначала использовать режим exact lookup внутри релевантного слоя и языка.

Для exact lookup:
- lexical / structured matching имеет приоритет;
- semantic candidates могут использоваться как backup, но не как основной механизм;
- `2026 norm` остаётся приоритетным слоем по умолчанию;
- `1995 deprecated` подключается только при явном основании.

Если exact lookup не дал прямого совпадения, это не даёт права сразу подменять ответ близкой по смыслу, но другой статьёй.

Сначала система должна:
- проверить альтернативные формы записи статьи / пункта;
- проверить соседние структурные элементы;
- только затем при необходимости использовать fallback-объяснение о недостаточности найденных данных.

---

## 21. MIXED-TOPIC И MULTI-ASPECT QUERY HANDLING

Если запрос содержит несколько независимых аспектов, retrieval не должен пытаться найти один универсальный фрагмент «на всё».

Нужно:
- разложить запрос по аспектам;
- искать подтверждение по каждому аспекту отдельно;
- затем передавать answer-layer либо несколько отдельных подтверждённых блоков, либо только подтверждённую часть вопроса.

Если один аспект подтверждён `norm`, а второй нет, retrieval не должен маскировать этот пробел `commentary` или тематически похожим фрагментом.

Если разные аспекты вопроса случайно попали в разные слои, они не должны автоматически сливаться в единый ответ без явной маркировки, какая часть на чём основана.

---

## 22. RETRIEVAL НЕ ДОЛЖЕН ДЕЛАТЬ СЛЕДУЮЩЕЕ

Retrieval не должен:
- поднимать `1995 deprecated` как текущий ответ по умолчанию;
- заменять `norm` разъяснением;
- скрывать deprecated-статус;
- скрывать project / transition сигналы;
- склеивать несопоставимые фрагменты;
- передавать answer-layer обезличенный текст без метаданных;
- полагаться только на semantic similarity;
- полагаться только на lexical similarity;
- считать `faq` доказательством существования нормы;
- подменять отсутствие подтверждения сильным score;
- выбирать слишком узкий контекст для широкого вопроса;
- использовать broad-query expansion как повод обещать полноту;
- использовать structural context expansion как повод подтягивать менее приоритетный слой;
- выбирать политическую рамку как критерий отбора;
- строить retrieval так, будто цель — подтвердить тезис пользователя.

---

## 23. СВЯЗЬ С ANSWER-LAYER

Retrieval и answer-layer должны работать как согласованная система.

Retrieval отвечает за то, чтобы в модель попадали:
- приоритетно правильные слои;
- явно маркированные фрагменты;
- достаточный структурный контекст;
- признаки deprecated / project / transition статуса;
- расширенная выдача для широких запросов;
- отсутствие искусственного смешения слоёв.

Answer-layer отвечает за то, чтобы:
- сначала отвечать по `norm`;
- не выдавать `commentary` и `faq` за норму;
- не маскировать слабый retrieval уверенным выводом;
- не делать политические выводы;
- не обещать полноту без отдельного основания;
- сохранять маркировку 1995 как `historical/deprecated` слоя.

Если retrieval передал неоднозначный или конфликтующий контекст, answer-layer должен это признавать, а не сглаживать.

---

## 24. SAFE FAILURE MODE

Если retrieval не может надёжно поддержать ответ, система должна предпочесть controlled under-answering, а не уверенную галлюцинацию.

Предпочтительное поведение:
- дать короткий safe fallback;
- предложить уточнить статью, тему или формулировку;
- при необходимости сузить вопрос;
- не компенсировать недостающий контекст общими знаниями или риторикой.

Лучше короткий ограниченный ответ, чем уверенная, но не подтверждённая реконструкция нормы.

---

## 25. ИТОГОВЫЙ ПРИНЦИП

При любом запросе retrieval должен действовать по приоритету:

1. Определить тип запроса.
2. Выбрать правильный слой и язык.
3. Защитить основной `2026 norm` от подмены `1995 deprecated`.
4. Использовать hybrid search, а не один вид похожести.
5. Для exact query сначала применять exact / lexical path.
6. Для широкого запроса расширять охват retrieval.
7. Для юридически неполного чанка подтягивать структурный контекст.
8. Сохранять `dataset`, `layer`, `status`, `article`, `point`, `deprecated` и другие значимые поля.
9. Не склеивать несопоставимые фрагменты.
10. Передавать answer-layer только такой контекст, который можно интерпретировать без скрытой подмены источника.

Хороший retrieval:
- возвращает приоритетно правильный слой;
- не допускает deprecated leakage;
- не теряет метаданные;
- поддерживает точные article lookup;
- расширяется для широких запросов;
- сохраняет различие между `norm`, `commentary`, `faq` и `historical/deprecated`;
- не провоцирует answer-layer на галлюцинации.

Плохой retrieval:
- выдаёт 1995 как текущую норму;
- подменяет `norm` commentary;
- теряет статус и происхождение фрагмента;
- режет юридически значимый контекст;
- даёт узкий случайный top-k на широкий запрос;
- позволяет semantic search победить правильный слой;
- склеивает разные темы и слои в один псевдоответ.
